# 会出海 Phase 2 产品路线图

日期：2026-07-05  
状态：建议路线图，基于 Phase 1 审核站验收结果整理

## 1. Phase 1 已完成能力摘要

Phase 1 已完成一个可审核、可演示、可部署的独立站 MVP：

- 审核站已可访问：`https://hch.ideaegg.com.cn`
- 首页、AI 顾问、询盘页、提交成功页、ops 预览页已打通。
- AI 顾问具备轻对话入口、咨询进度、轻摘要、阶段判断和配置页承接。
- 服务端已具备 Phase 1 Agent route 和结构化 `AgentTurnResult`。
- 方案配置支持经济型 / 标准型 / 高配型、服务项调整和预算结构估算。
- 资源主档与当次询价单的两层模型已在类型、mock 数据和 ops 页面中落地。
- 客户字段白名单和内部字段保护测试已建立。
- 审核站保持 `noindex / nofollow`，不是正式生产发布。

Phase 1 仍然不是正式业务闭环：

- AI provider 未接真实生产能力。
- 资源、报价、档期、供应商响应、线索和通知仍有 MOCK / 示例内容。
- ops 权限仍是审核用最小保护，不是正式登录与权限体系。
- 数据未持久化到真实业务数据库。
- 正式品牌矢量资产、真实图片和真实案例仍需补齐。

## 2. Phase 2 总目标

Phase 2 的目标是从“可审核 MVP”进入“真实业务闭环”：

1. 客户能够自然咨询并留下结构化需求。
2. AI 能基于真实知识和资源主档给出可靠的方向建议与预算解释。
3. 运营能够接收线索、补充信息、发起当次询价并更新报价状态。
4. Chris / 运营能够收到提醒并完成跟进。
5. 客户侧始终只看到客户可见字段和参考范围，不泄露内部判断。
6. 站点具备正式上线、正式收录和多渠道入口扩展条件。

## 4. 详细路线图

### 4.1 真实 AI provider 接入与对话评估

业务价值：
- 让 AI 顾问从规则型 / MOCK 体验升级为可持续优化的真实咨询能力。
- 提升模糊咨询、城市比较、活动方案建议和预算解释质量。

前置条件：
- 明确可使用的 AI provider、模型、成本预算和调用边界。
- 整理城市知识卡、活动类型知识卡、FAQ、服务边界和禁止承诺清单。
- 明确生产环境 API key 管理方式。

建议验收标准：
- 每轮服务端返回结构化 `AgentTurnResult`，包含 stage、understoodIntent、extractedFacts、answerStrategy、reply、followupQuestion、canEnterConfigurator、shouldNotifyOperator。
- 对模糊咨询先回答问题，再最多追问一个关键问题。
- 不把“投资大会”等具体意图压成泛化“商务会议”。
- 未到 `structuring` 不进入配置页。
- 到 `handoff_ready` 可生成内部提醒，但客户侧不泄露内部判断。
- 建立至少 20 条真实业务对话评估样本。

风险 / 注意事项：
- 不允许模型直接承诺正式价格、实时档期、付款和取消条款。
- 不允许模型输出内部字段、内部评分或推荐跟进话术给客户。
- 需要日志和评估机制，否则无法稳定优化顾问质量。

必须等 Chris 提供的资料：
- 常见客户问法、真实销售口径、城市差异、预算解释方式、禁止承诺清单。

### 4.2 会务资源库数据库化

业务价值：
- 把资源从页面 MOCK 变成可维护的运营资产。
- 支撑 AI 顾问和 ops 询价流程使用同一套资源主档。

前置条件：
- 确认 `resource_master` 字段、枚举和可见性边界。
- 补齐真实资源的供应商、城市、容量、服务范围、适用场景和参考价区间。
- 确定资源内容素材状态：`draft / needs_review / verified / public_ready`；`published` 留到 Phase 3 具备正式发布系统后再使用。

建议验收标准：
- Supabase 或等价数据库中可新增、编辑、停用资源。
- 客户侧只消费经过白名单映射、已核对且 public-safe 的资源字段；内容素材状态不直接暴露给客户。
- `referencePriceMin / referencePriceMax` 被明确标记为参考范围，不是正式报价。
- 内部字段如 supplierName、internalNegotiationNote、internalRiskNote 不出现在客户响应中。

风险 / 注意事项：
- 不要把某次临时报价回写覆盖资源主档。
- 资源主档不是固定价格库，只能沉淀参考区间、适用条件和动态影响因素。

必须等 Chris 提供的资料：
- 真实资源清单、合作状态、参考区间、淡旺季规则、最低消费、提前期和内部合作备注。

### 4.3 AI 对话、线索、需求摘要持久化

业务价值：
- 客户刷新页面或跨设备继续咨询时，不丢失上下文。
- 运营可以查看完整线索来源、需求摘要和跟进状态。

前置条件：
- 确认 `inquiries`、`ai_conversations`、`ai_operator_alerts` 的字段边界。
- 明确匿名访客、表单提交客户和后续登录客户之间的关联策略。
- 确定数据保留周期和隐私说明。

建议验收标准：
- 对话消息、结构化摘要、stage、canEnterConfigurator 和 handoff 状态可持久化。
- 询盘表单提交后可关联前置 AI 咨询内容。
- 客户侧刷新后仍能看到自己的轻摘要和下一步。
- ops 可看到线索摘要，但客户侧不暴露运营评分。

风险 / 注意事项：
- 原始对话可能包含个人联系方式，需要最小化展示与权限控制。
- 不要把 server_only 和 operator 字段返回给客户侧。

必须等 Chris 提供的资料：
- 运营需要看到的线索字段、跟进状态、内部备注规范。

### 4.4 Chris / 运营提醒机制

业务价值：
- 把 AI 咨询转化为可跟进线索，减少漏单。
- 明确谁在什么时候接手，提升响应速度。

前置条件：
- 明确提醒渠道：邮件、WhatsApp、企业微信、飞书或短信。
- 明确接收人、值班机制、SLA 和升级规则。
- 明确哪些情况触发提醒。

建议验收标准：
- 到达 `handoff_ready` 或客户提交询盘后生成内部提醒。
- 提醒包含客户可跟进摘要、活动关键信息、缺失信息和建议下一步。
- 提醒不发送给客户，不包含可被客户误解为正式报价的内容。
- 提醒发送状态可追踪。

风险 / 注意事项：
- 推荐开场白、线索优先级、风险判断只能内部可见。
- 通知失败需要重试和人工兜底。

必须等 Chris 提供的资料：
- 接收人、联系方式、服务时间、线索分配规则、响应承诺。

### 4.5 Ops 登录与权限体系

业务价值：
- 让内部资源、线索、报价、供应商响应和谈判备注有正式访问边界。
- 替代审核阶段 token 保护。

前置条件：
- 明确账号体系：邮箱登录、OAuth、企业微信或 Supabase Auth。
- 明确角色：Chris、运营、内容维护、只读审核。
- 明确每个角色可访问的页面和字段。

建议验收标准：
- 未登录无法访问 `/ops/*`。
- 不同角色看到不同页面和操作入口。
- 内部字段只在授权 ops 页面出现。
- 登录、退出、会话过期和权限不足状态清楚。

风险 / 注意事项：
- 不能依赖前端隐藏字段做权限。
- 导出、截图、分享链接也需要考虑内部字段泄露风险。

必须等 Chris 提供的资料：
- 内部团队成员、角色分工、外部协作人员是否需要只读权限。

### 4.6 报价、询价、档期流程细化

业务价值：
- 把“参考预算”推进到“本次活动可执行报价确认”。
- 让资源主档和当次询价单形成真实闭环。

前置条件：
- 确认 `inquiry_quote_requests` 字段、状态机和审批流程。
- 明确供应商询价方式、报价有效期、付款条款和取消条款。
- 明确哪些信息可以回显给客户，哪些只能内部查看。

建议验收标准：
- 可从线索或资源发起当次询价。
- 可更新档期状态、询价状态、报价范围、有效期、付款和取消条款。
- 客户侧只在本次询价确认后看到客户可见摘要。
- 过期报价不再作为有效客户承诺。

风险 / 注意事项：
- 资源主档不能被当次报价覆盖。
- 供应商原始响应、冲突原因、谈判空间和底价不能对客户展示。

必须等 Chris 提供的资料：
- 实际询价流程、供应商沟通模板、报价审批规则、合同与付款条款口径。

### 4.7 GEO 内容素材上游采集与候选池

业务价值：
- 让 Chris / 运营在维护真实资源时同步沉淀城市、场景、FAQ、案例和图片授权素材。
- 为后续 GEO / SEO 内容矩阵提供可信上游，而不是临时编写公开页面。

前置条件：
- 资源主档字段能区分运营内部信息和 public-safe 候选素材。
- 图片、案例和 FAQ 素材具备授权状态或审核状态。

建议验收标准：
- `/ops/resources` 有独立的公开内容素材分区。
- `/ops/content-candidates` 能汇总候选素材并标出缺口。
- 客户侧页面不出现内容候选状态、供应商内部信息、报价冲突或运营判断。

风险 / 注意事项：
- Phase 2 不做公开页面自动生成，不改 sitemap，不开启正式收录。
- `public_ready` 只代表内部素材已准备好进入后续内容生产，不等于已经公开发布。

必须等 Chris 提供的资料：
- 可公开资源摘要、城市经验、典型客户问题、案例授权、图片授权和内容审核口径。

### 4.8 GEO / 内容矩阵和正式收录策略

业务价值：
- 为正式上线后的搜索与 AI 搜索可见性做准备。
- 承接客户对城市、活动类型、预算和服务问题的主动搜索。

前置条件：
- 确认正式域名和是否允许搜索引擎收录。
- 补齐真实 FAQ、城市内容、案例内容和服务边界。
- 清理或替换所有 MOCK 内容。

建议验收标准：
- `NEXT_PUBLIC_SITE_INDEXABLE=true` 只在正式上线后开启。
- robots 和 sitemap 输出正式 URL。
- 内容页面覆盖城市、场景、服务项、FAQ 和案例。
- 每页有清晰来源、更新时间和非承诺性报价说明。

风险 / 注意事项：
- 审核站不能被提前收录。
- 未确认真实资源和价格前，不做正式 SEO 承诺。

必须等 Chris 提供的资料：
- FAQ、真实案例、城市资源、正式联系方式、可公开图片和客户授权材料。

### 4.9 微信小程序规划

业务价值：
- 承接中国客户移动端咨询、转发和后续跟进。
- 让 AI 顾问成为常驻轻入口，而不只是网页功能。

前置条件：
- 明确小程序主体、认证、类目、合规要求和登录方式。
- 明确网页与小程序之间的会话同步策略。
- 确认是否接入微信客服、订阅消息或企业微信。

建议验收标准：
- 小程序支持轻咨询、需求摘要、询盘提交和顾问跟进状态。
- 与 Web 共用服务端 Agent、资源主档和线索数据。
- 客户侧字段边界与 Web 一致。

风险 / 注意事项：
- 小程序审核、消息触达和隐私协议需要提前准备。
- 不要在小程序内展示 ops 内部字段或供应商内部信息。

必须等 Chris 提供的资料：
- 小程序主体信息、服务类目、客服机制、是否需要微信支付或订阅消息。

### 4.9 正式品牌矢量 logo 与品牌规范

业务价值：
- 把当前批准 PNG interim 资产升级为可长期使用的品牌系统。
- 支撑正式官网、提案、名片、物料和小程序图标。

前置条件：
- 设计线程输出最终矢量稿和使用规范。
- 明确主标、中文辅助、反白版、安全区、最小尺寸和禁用规则。

建议验收标准：
- 交付 SVG / PDF / PNG 多格式品牌资产。
- Header、footer、favicon、app icon、小程序 icon 使用正式资产。
- 文档记录颜色、字体、间距和反白使用规则。

风险 / 注意事项：
- 不再自行重画 logo，不探索新方向。
- 必须避免 IOI 误读、宗教感、徽章感、旅游社气质和 mascot 化。

必须等 Chris / 设计线程提供的资料：
- 最终矢量 logo、品牌色规范、字体规范、实际应用场景需求。

## 5. 必须等真实资料后才能继续的事项

以下事项不建议继续用 MOCK 深做：

- 真实资源库发布：需要资源名称、合作方、容量、参考区间、适用条件、联系方式和内容授权。
- 正式报价口径：需要实际询价流程、报价有效期、付款、取消、税费和币种规则。
- 正式 FAQ / GEO 内容：需要 Chris 的真实回答、城市经验和典型客户问题。
- 真实案例展示：需要客户授权、图片授权、活动规模、成果描述和可公开范围。
- 通知机制：需要运营接收人、渠道、SLA 和升级规则。
- 正式 SEO 收录：需要最终域名、真实内容、正式联系方式和 MOCK 清理。
- 小程序立项：需要主体、类目、客服和隐私合规材料。

## 6. 建议实施节奏

### Phase 2.1：真实线索与 Agent 基座

- 接真实 AI provider。
- 建立对话评估样本。
- 持久化 AI 对话、需求摘要和询盘。
- 上 ops 登录与权限。

验收目标：客户咨询能被可靠保存，运营能安全查看并接手。

### Phase 2.2：资源与询价闭环

- 资源主档数据库化。
- 当次询价单状态机细化。
- Chris / 运营提醒机制上线。
- 供应商响应、报价范围、有效期、付款和取消条款进入内部流程。

验收目标：从客户需求到运营询价、报价确认和客户可见摘要形成闭环。

### Phase 2.3：正式发布准备

- GEO / 内容矩阵。
- 正式品牌矢量资产。
- 正式域名、正式收录和 sitemap。
- 微信小程序方案与立项。

验收目标：从审核站切换为可长期运营的正式站点与多渠道入口。

## 7. 边界提醒

- 当前审核站已经上线，但仍不是正式生产完成。
- 客户侧只能看到参考范围和需顾问确认事项，不能看到内部评分、优先级、风险和谈判备注。
- 资源价格是动态浮动的，系统应维护资源主档和当次询价单，不应沉淀固定长期成交价。
- Phase 2 的关键不是继续堆页面，而是接入真实资料、真实数据、真实权限和真实运营流程。
